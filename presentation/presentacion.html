1. El Problema de la Mochila Extendido

Formulación Clásica

Maximize Z = Σ vᵢxᵢ
s.a. Σ wᵢxᵢ ≤ W,   xᵢ ∈ {0, 1}

Problema NP-hard: no existe algoritmo exacto polinómico. Los algoritmos genéticos ofrecen soluciones de buena calidad en tiempo razonable.

Aplicaciones Reales

  • Inversión de activos
  • Corte de materiales
  • Carga de contenedores
  • Asignación de recursos

Restricciones de la Versión Extendida

Dura Peso máximo — Σ wᵢxᵢ ≤ W

Dura Volumen máximo — Σ volᵢxᵢ ≤ V

Dura Incompatibilidades — pares prohibidos

Dura Dependencias — ítems asociados

Blanda Categorías — min/máx por grupo

Función de Aptitud

fitness(X) = Σvᵢxᵢ − α·peso − β·vol − γ·cat − δ·incomp − ε·deps

¿Por qué 0.3% factible?

Con 499 incompatibilidades y 10,000 ítems, el GA selecciona ~3,000 ítems generando violaciones inevitables. La "death penalty" (δ=1,000,000) hace que el fitness sea muy negativo para soluciones infeasibles.

Shared Memory

10,000 ítems × 4 arrays × 4 bytes = 160KB > 48KB límite. Fallback automático al kernel básico.

¿Por qué CUDA es más rápido?

CPU: 1 núcleo evalúa 4,096 individuos secuencialmente
↓
GPU: 2,816 cores evalúan 4,096 individuos en paralelo

La evaluación de fitness es el cuello de botella. Con 1,000 ítems, cada individuo tarda ~8μs en CPU vs ~0.2μs en GPU.

Calidad Consistente

CPU y CUDA producen soluciones de igual calidad

9. Análisis Técnico

✅ Optimizaciones que Funcionaron

  • Shared memory: Reutilización de datos de items dentro del bloque
  • Inicialización greedy: +5% factibilidad vs aleatorio puro
  • Block size 128: Mejor occupancy en RTX 3060
  • Población en GPU: Minimiza transferencias host↔device
  • CUDA events: Medición precisa sin overhead

❌ Optimizaciones con Limitaciones

  • Shared memory en large: 160KB > 48KB (fallback automático)
  • Block size 512: 35% más lento que 128 (menor occupancy)
  • curand: Calidad inferior a RNG en CPU

10. Fragmentos de Código

Kernel de Fitness (Básico)

__global__ void kernelFitnessBasic(
    int* d_fitness, int* d_valor_total,
    bool* d_factible, const int* d_genes,
    const int* d_item_valores,
    const int* d_item_pesos,
    int pop_size, int n_items,
    int capacidad_peso, ...) {

  int idx = blockIdx.x * blockDim.x
            + threadIdx.x;
  if (idx >= pop_size) return;

  int valor=0, peso=0, vol=0;
  for (int i = 0; i < n_items; i++) {
    if (d_genes[idx*n_items + i]) {
      valor += d_item_valores[i];
      peso  += d_item_pesos[i];
      vol   += d_item_volumenes[i];
    }
  }
  // Penalizaciones y factibilidad...
  d_fitness[idx] = valor - penalizacion;
}

Kernel Optimizado (Shared Memory)

__global__ void kernelFitnessOptimized(
    ..., int n_items, ...) {

  extern __shared__ int shared_mem[];
  int* s_valores = shared_mem;
  int* s_pesos = shared_mem + n_items;

  // Carga coalescente a shared memory
  for (int i = threadIdx.x;
       i < n_items; i += blockDim.x) {
    s_valores[i] = d_item_valores[i];
    s_pesos[i]   = d_item_pesos[i];
  }
  __syncthreads();

  // Fitness usando shared memory
  for (int i = 0; i < n_items; i++) {
    if (my_genes[i]) {
      valor += s_valores[i];
      peso  += s_pesos[i];
    }
  }
}

11. Conclusiones

Logros Principales

  • Speed-up de hasta 49x en medium (1,000 ítems)
  • Speed-up de 36x en large (10,000 ítems)
  • Calidad consistente entre CPU y CUDA
  • 3 variantes funcionales con 270 experimentos
  • GPU mantiene población completa durante evolución

📚 Aprendizajes

  • Paralelizar fitness es la optimización más impactante
  • Shared memory es efectiva solo para instancias pequeñas
  • Block size 128 óptimo para RTX 3060
  • Escalado de penalizaciones es crítico
  • Inicialización greedy mejora factibilidad
