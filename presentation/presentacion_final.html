El Problema de la Mochila Extendido

Formulación

$$\text{Maximizar } Z = \sum_{i=1}^{n} v_i x_i$$

sujeto a

$$\sum w_i x_i \leq W,\quad \sum vol_i \cdot x_i \leq V$$

NP-hard  →  Algoritmos Genéticos

Aplicaciones Reales

  • Inversión de activos financieros
  • Corte óptimo de materiales
  • Carga de contenedores logísticos
  • Asignación de recursos computacionales

5 Restricciones de la Versión Extendida

Dura Peso máximo $\ \sum w_i x_i \leq W$

Dura Volumen máximo $\ \sum vol_i\,x_i \leq V$

Dura Incompatibilidades entre pares de ítems

Dura Dependencias obligatorias entre ítems

Blanda Categorías (mínimo/máximo por grupo)

Función de Aptitud

$$f(X) = \sum v_i x_i - \alpha W - \beta V - \gamma C - \delta I - \varepsilon D$$

Death penalty: $\delta = \varepsilon = 1{,}000{,}000$ para restricciones duras

Kernel Fitness (Básico)

__global__ void kernelFitnessBasic(
    int* d_fitness, ..., int n_items) {

  int idx = blockIdx.x * blockDim.x
            + threadIdx.x;
  if (idx >= pop_size) return;

  int valor=0, peso=0, vol=0;
  for (int i = 0; i < n_items; i++) {
    if (d_genes[idx*n_items + i]) {
      valor += d_item_valores[i];
      peso  += d_item_pesos[i];
      vol   += d_item_volumenes[i];
    }
  }
  // Penalizaciones y factibilidad...
  d_fitness[idx] = valor - penalizacion;
}

1 hilo por individuo. Acceso coalescente a memoria global.

Kernel Fitness (Opt — Shared Memory)

__global__ void kernelFitnessOptimized(
    ..., int n_items, ...) {

  extern __shared__ int shared_mem[];
  int* s_valores = shared_mem;
  int* s_pesos = shared_mem + n_items;

  // Carga coalescente a shared mem
  for (int i = threadIdx.x;
       i < n_items; i += blockDim.x) {
    s_valores[i] = d_item_valores[i];
    s_pesos[i] = d_item_pesos[i];
  }
  __syncthreads();

  // Fitness usando shared memory
  for (int i = 0; i < n_items; i++) {
    if (my_genes[i])
      valor += s_valores[i];
  }
}

Shared memory para reutilización. Fallback automático si >48KB.

Análisis Técnico

✅ Qué Funcionó

  • Shared memory (5–10% mejora)
  • Inicialización greedy (+5% factibilidad)
  • Block size 128 (mejor occupancy)
  • Población en GPU (minimiza transfers)
  • CUDA events (medición sin overhead)
  • Fallback automático para large

❌ Limitaciones

  • Shared mem en large (160KB > 48KB)
  • Block size 512 (35% más lento)
  • Factibilidad baja en large (0.3%)
  • Elitismo requiere transfer CPU
  • Curand calidad inferior a RNG CPU

📊 Calidad vs Tiempo

  • Pop=1,024: rápido, calidad base
  • Pop=4,096: óptimo balance
  • Pop=16,384: +3–5% calidad, 4× tiempo
  • Más generaciones > más población
